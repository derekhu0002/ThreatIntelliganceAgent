


在一个现代的 LLM 应用架构中，**Agent（智能体）+ 多 Skill（技能/SOP）+ 多 Tool（工具/API）** 是一种非常经典且高扩展性的设计模式。

为了方便理解，我们可以用一个现实中的比喻：
*   **Agent（智能体）**：是**大厨**（大脑），负责听取顾客需求、思考怎么做、安排先后顺序。
*   **Skill（技能）**：是**菜谱**（SOP/标准作业程序），告诉大厨面对某道具体的菜时，第一步做什么、第二步做什么、用什么火候。
*   **Tool（工具）**：是**厨具和食材库**（API/执行器），比如微波炉、切菜机、冰箱，是真正和外部物理世界/数据源交互的抓手。

下面我将详细拆解这三者分别应该如何设计，以及如何将工具高效匹配给 Agent。

---

### 一、 Agent 中需要描述什么？（核心是大脑与调度器）

Agent 的核心定义文件（如 `Agent.md` 或 `config.yaml`）不需要写死具体的业务细节，而是要定义其**行为准则、调度逻辑和边界**。

一般需要包含以下内容：
1. **身份与人设 (Identity & Persona)**
   *   **描述**：你是谁？你的总体目标是什么？
   *   *示例*：“你是高级威胁情报分析师，负责安全事件的意图识别、证据收集与研判。”
2. **意图路由逻辑 (Intent Routing & Planning)**
   *   **描述**：当收到用户输入时，你的思考框架是什么？（比如 ReAct 框架：Thought -> Action -> Observation）。
   *   *示例*：“收到输入后，你必须先分析用户的意图，从可用的 Skill 列表中检索出最匹配的 Skill，然后严格按照该 Skill 的步骤执行。”
3. **能力与权限控制 (Permissions & Constraints)**
   *   **描述**：你能做什么，绝对不能做什么（防止幻觉和越权操作）。
   *   *示例*：“你只能使用被授权的工具；禁止猜测数据库字段名；涉及数据修改必须请求人工确认。”
4. **大模型配置 (LLM Configuration)**
   *   **描述**：选用的模型基座、Temperature（温度值，推理场景通常设为 0~0.2 以保证严谨性）、Top-P 等参数。

---

### 二、 SKILL 中需要描述什么？（核心是SOP与业务逻辑）

Skill 的核心是将复杂的模糊任务转化为**可执行的标准作业程序 (SOP)**。每新增一个业务场景，只需新增一个 Skill，而不需要改动 Agent 的底层代码。

一个高质量的 Skill（如 `SKILL.md`）一般包含：
1. **元数据与触发条件 (Metadata & Triggers)**
   *   **描述**：Skill 的名称、功能摘要，以及“在什么情况下 Agent 应该选择这个 Skill”。这部分是写给 Agent 里的意图识别路由看的。
   *   *示例*：`name: ioc_quick_check`, `description: 当用户输入IP、域名或HASH并询问其安全性时触发此技能。`
2. **槽位与前置依赖 (Slots & Prerequisites)**
   *   **描述**：执行这个技能必须从用户输入中提取到哪些关键信息？如果没有，Agent 应该反问用户补充。
   *   *示例*：必须提取到 `target_ioc` 和 `ioc_type`。
3. **标准作业步骤 (SOP - Standard Operating Procedure)**
   *   **描述**：分步骤指导 Agent 如何思考以及如何调用工具。这是 Skill 的灵魂。
   *   *示例*：
       *   *Step 1: 先调用工具 A 获取数据源目录。*
       *   *Step 2: 调用工具 B 获取目标 IOC 的信誉分。*
       *   *Step 3: 如果分数 < 50，则结束并返回安全；如果 > 50，再调用工具 C 获取历史攻击上下文。*
4. **输出规范 (Output Format)**
   *   **描述**：最终交付给用户的内容结构。
   *   *示例*：“输出必须包含：1. 恶意定性；2. 置信度；3. 关联的APT组织；4. 处置建议，并使用 JSON 或指定的 Markdown 模板返回。”

### 三、 如何给 Agent 匹配工具？（Tool Matching & Binding）

如何让 Agent 在正确的时候使用正确的 Tool？通常有三种设计层次，由浅入深：

#### 1. 静态绑定（全局可见模式）
最基础的做法，直接在 Agent 的配置/代码中声明它能看到哪些工具。
*   **做法**：在 Agent 的初始化中，把 工具库中对应的 JSON Schema（也就是 OpenAI Function Calling 格式）全部塞入 Agent 的 System Prompt。
*   **优点**：实现简单，Agent 随时可以调用任何工具。
*   **缺点**：当工具有几十上百个时，Prompt Context 会爆掉，且 LLM 极易产生幻觉，选错工具或传错参数。

#### 2. 按需动态加载（Skill 驱动模式 / 最推荐做法）
为了解决工具过多的问题，工具的分配权应该交给 **Skill**，Agent 只充当执行通道。
*   **设计机制**：
    1. Agent 启动时，只挂载 1 个核心路由工具（比如 `search_skills`）。
    2. Agent 收到任务 -> 调用 `search_skills` 匹配到对应的 `Skill_X`。
    3. `Skill_X` 被加载到上下文中，Skill 中明确声明了：“完成本技能，你需要且只能加载 `Tool_A` 和 `Tool_B`”。
    4. 平台底层拦截到这个信号，动态将 `Tool_A` 和 `Tool_B` 的 Function Schema 注入给 Agent。
*   **优点**：按需分配，Prompt 极度干净，极大地降低了 LLM 的幻觉率。

#### 3. 泛型/多态工具设计（Tool 自身的抽象）
如同我们在前面例子中看到的 `ai4x_query`，这也是一种高级设计理念：**减少工具数量，增加工具内部的参数多态性**。
*   **做法**：不设计 `query_neo4j`、`query_mongodb`、`query_cve` 10 个不同的工具，而是只设计一个泛型工具 `unified_query_tool`。
*   **参数映射**：
    *   `command`: 告诉工具是查目录、查Schema、还是查数据。
    *   `sourceId`: 告诉工具路由到哪个底层组件。
*   **匹配方式**：Agent 永远只需要绑定这 1 个泛型工具，但在 Skill 的 SOP 中，详细指导 Agent 针对不同业务传入不同的 `sourceId` 和 `command` 参数。

### 总结：完整的工作流体验

1. **用户输入**：“帮我查一下 CVE-2023-1234 的影响。”
2. **Agent 思考**：我是一个情报分析师，用户的意图是“漏洞影响评估”。我需要调取《漏洞影响评估》这个 **Skill**。
3. **加载 Skill**：Agent 阅读该 Skill 的 SOP，得知第一步需要查询漏洞库。同时，框架动态赋予了 Agent 使用 `cve_query_tool` 和 `graph_query_tool` 的权限。
4. **Agent 使用 Tool**：
   *   Agent 组装 JSON 格式的参数 `{"cve_id": "CVE-2023-1234"}`。
   *   触发底层的 Function Calling 调用 `cve_query_tool`，获得受影响的组件名。
5. **按 Skill 继续推进**：Agent 根据 SOP 第二步，拿到组件名后，再次调用 `graph_query_tool` 查询内部架构图谱。
6. **Agent 总结**：依据 Skill 规定的**输出规范**，将工具返回的数据整合成人类可读的报告，返回给用户。

这种 **“Agent 管决策控制流，Skill 管业务逻辑流，Tool 管数据操作流”** 的铁三角架构，是目前业界处理复杂 B 端大模型业务最成熟的范式。
